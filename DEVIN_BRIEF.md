3. **Build/assembly animation.** Add the skeleton→finished timeline (transform/opacity/clip-path only), gated behind `prefers-reduced-motion` and behind a JS feature-detect so the no-JS finished state from step 2 remains the fallback.

Each step above should be its own PR/commit — this repo has no CI to catch regressions automatically, so small reviewable diffs are the only mitigation available.

### Specific risks in this codebase

- **No component boundary today** means it's easy to accidentally duplicate the hero (copy-pasting into `about.blade.php` too) instead of scoping the change to the homepage only. Explicitly confirm with the user whether this two-column redesign is homepage-only (as the request implies) before touching `about.blade.php`.
- **`.animate-on-scroll` global IntersectionObserver** ([resources/js/app.js:126](resources/js/app.js:126)) queries `document.querySelectorAll('.animate-on-scroll')` on `DOMContentLoaded` site-wide. If the new hero reuses this class for any element, it will be swept into that observer; if it doesn't but a class name collides with new hero-specific classes, check for that before naming new utility classes (e.g. don't reuse `.visible` generically).
- **Sticky nav height coupling**: the hero's height is `h-[calc(100svh-4rem)]`, hard-coding the `h-16` (4rem) sticky nav height from `layouts/app.blade.php:52`. If the two-column layout needs more vertical room for the right-column preview on smaller viewports, changing this calc or adding `min-h-*` (as `about.blade.php` does with `min-h-[36rem]`) is safe and precedented — just don't decouple it from the nav height silently.
- **`overflow-hidden` on the hero `<section>`**: currently used to clip the background video/gradients. If the build-animation's skeleton pieces transform in from outside their container (e.g. `translateX` off-screen), verify they're clipped correctly by this same `overflow-hidden`, or the assembly animation could visibly overflow into the nav or the next section on some viewports.
- **External video dependency**: the current hero streams its background video from `assets.mixkit.co` (third-party CDN, not a local asset). This has nothing to do with the new right column, but if the left-column background/overlay treatment is retained behind or around the new grid, confirm whether the video stays, is removed, or is scoped only to a smaller left-column background — the request doesn't say, and silently dropping it changes visual identity.
- **Zero automated regression coverage.** No CI, no visual regression tool, no JS tests. The "zero CLS" and "40 KB gzipped" constraints are easy to silently violate without a human explicitly measuring them (Lighthouse/DevTools Performance + Network panel) after each step — build this into the sequencing above rather than checking once at the end.
- **Portfolio Cinematic section immediately below** ([home.blade.php:186](home.blade.php:186)) is a `h-[500vh]` sticky-scroll-hijacking section. It's unrelated to the hero but sits directly after it in the same file — when editing/reordering `home.blade.php`, be careful not to disturb its `id="portfolio-cinematic"` anchor or the scroll-math that depends on the hero's height (skim its script for any hero-height assumptions — a quick read shows it computes progress from its own section's `scrollTop`/`offsetTop`, not the hero's, so it should be safe, but verify after step 1's extraction).

### Constraint conflicts with current architecture, and recommendations

- **"Reuse existing … design tokens" vs. "different typeface pairing per theme"**: the constraint says reuse tokens/no new dependencies, but the feature asks for a *different* typeface pairing per theme chip. Current `fontFamily` tokens (`sans`/`heading`/`display`/`logo`) only define **two** typefaces total (Manrope, Outfit, plus Playfair Display for display). Three themes needing genuinely distinct pairings will either (a) reuse combinations of the existing 3 families in different roles per theme (recommended — zero new dependency, e.g. restaurant leans on `font-display` (Playfair) for headings, construction leans on `font-heading` (Outfit) bold/uppercase, hairdresser leans on `font-sans` (Manrope) light-weight), or (b) require loading 1-2 additional Google Fonts, which conflicts with "no new dependencies without justification." **Recommendation: use option (a)** — re-pair the three existing families per theme rather than adding fonts. This is very achievable and keeps the font `<link>` in `layouts/app.blade.php:15-16` untouched.
- **"Transform and opacity only, except a clip-path or scaleX text mask" vs. the skeleton→finished visual change**: a wireframe-to-mockup transition typically implies content/background swaps (skeleton gray blocks → colored image blocks, placeholder lines → real-looking text), which is a `background`/`content` change, not transform/opacity. **Recommendation**: build the "finished" DOM as the only DOM that ever exists (correct for zero-CLS and no-JS-fallback anyway), and have the "skeleton" state be the *same* elements with `background-color: theme gray`, `border-radius`, reduced opacity, and `scaleX`/`scaleY` transforms (e.g. text lines start at `scaleX(0.3)` and grow, image blocks start desaturated/gray via `opacity` cross-fade between a gray-block pseudo-layer and the real content layer rather than swapping `background-image`). This keeps every animated property within transform/opacity/clip-path and satisfies "assembles from a skeleton" without ever mutating layout-affecting properties (avoiding both CLS and the transform-only constraint). This is the single most architecturally important decision in the brief — flag it to the user if any other approach seems easier, because most "skeleton loader" implementations in the wild use non-compliant properties (width/height/background-image swaps).
- **"Feature flag rendering a static variant for A/B testing" vs. "no CI/no experimentation framework"**: there's no A/B testing infra (no GrowthBook, Optimizely, etc. found in `composer.json` or `package.json`). The config-boolean approach recommended above gives you a manual flag, not statistically-assigned bucketing. If true randomized A/B assignment (not just an on/off toggle you flip manually) is required, that's a materially bigger scope addition (cookie-based bucketing, an analytics event, a stats service) — **confirm with the user whether "feature flag" means (a) a kill-switch/manual toggle for safe rollout, which is what this brief scopes, or (b) a real randomized experiment with metrics**, before Devin builds either.
- **40 KB gzipped budget**: not in conflict with anything, but note it's generous relative to what's needed — a transform/opacity-only timeline driven by vanilla JS (no animation library, per the inventory above) for a handful of elements should land well under 10 KB gzipped. If Devin's implementation approaches 40 KB, that's a signal something (e.g. per-theme content data inlined as JS objects, or SVG icon sets) should move to Blade-rendered markup instead of JS-constructed DOM.

## Step 4 — Where I disagree / think this is the wrong use of effort

- **The transform/opacity-only constraint plus a genuine wireframe→finished "assembly" illusion is the hardest part of this spec, and it's easy to under-scope.** A skeleton-to-mockup transition that looks convincing (not just "boxes fade in") usually wants staggered reveals, easing curves tuned per element, and per-theme content substitution — all while never touching `width`/`height`/`background-image` directly. This is achievable (see my recommendation above: same DOM, gray overlay layer cross-fades via opacity, text lines scale via `scaleX`), but it's meaningfully more design/animation-craft work than "replace hero, add two columns" suggests at a glance. If the 2.5s build sequence ends up looking like generic shimmer-loading rather than a deliberate "website being built" narrative, that's most of the perceived value of this feature lost — I'd push back on treating this as a routine layout task and budget real design-review time (not just code-review time) for the animation itself, ideally with the user looking at an in-browser WIP before Devin polishes all three themes.
- **Three fully-distinct themes (color + typography + imagery + layout arrangement) is closer to building a tiny content/theming system than a single "hero variant."** Given there's no existing multi-theme infrastructure in this codebase (Tailwind config has one fixed palette, one font set), this task is implicitly asking Devin to invent a small theming pattern from scratch with no local precedent to follow. That's fine, but I'd flag it's the second-largest scope item after the animation itself, and "different body block arrangement" per theme (not just recoloring) means three separate mini-layouts to build and maintain in the same file — worth confirming the user wants three genuinely distinct layouts vs. one layout with cosmetic per-theme swaps, since the former is meaningfully more code to keep in sync.
- **The A/B feature-flag constraint, as written, may be solving a problem this codebase isn't set up to measure.** There's no analytics/experimentation infrastructure here at all (no GrowthBook, no PostHog, no GA event wiring found in the repo). A flag that renders a static variant is easy to build, but without a way to *measure* which variant converts better, "for A/B testing" is aspirational — the flag will function as a rollout kill-switch in practice, not an experiment. I'd say that's still worth building (cheap, safe), but I'd tell the user explicitly that real A/B measurement is a separate, unscoped piece of work if they want actual experiment results, not just the ability to toggle.
- **Net effort assessment**: given the no-CI, no-lint, no-existing-component, no-animation-library starting point, this is a bigger lift than the six-bullet constraint list makes it look — closer to "design and build a small themeable animation micro-system inside a Blade monolith" than "swap the hero." That's not a reason not to do it, but I'd set expectations with the user that steps 2-4 in the sequencing above (static layout → animation → theming) each deserve their own review pass, not a single "hero redesign" PR.
